# Twrminal v0.2.0 — Tags, Tag Memories, Tag Defaults

v0.2 adds one organizational primitive on top of v0.1: **tags**.
Tags carry memories (markdown injected into the system prompt
whenever they're attached to a session), plus optional default
working directory and default model for new sessions. There is no
separate "project" concept — a pinned tag like `twrminal` IS the
project.

Design principle: memories are **pointers and rules**, not mirrored
document libraries. Claude uses its normal file tools to open
directories when it needs to. Keep the system simple.

---

## Summary of changes

- New primitive: **tags** — global, named, with `pinned` +
  `sort_order`. Many-to-many with sessions via `session_tags`.
- **Tag memories** — per-tag markdown injected into the system
  prompt when that tag is on a session. Always injected when
  present.
- **Tag defaults** — optional `default_working_dir` and
  `default_model` per tag. The new-session form pre-fills from the
  highest-precedence attached tag.
- **Layered system prompt** — base + tag memories + session
  override, deterministically assembled and inspectable.
- **Every session has ≥1 tag.** The new-session form is gated on
  at least one tag; the API rejects tag-less session creation.
- No breaking changes to the WS protocol or conversation flow.

## Migrations

Tags ship as a sequence of small migrations so each build step
lands independently. Per memory `shipping_cadence.md` each slice
gets its own tag + push.

### `0006_tag_primitives.sql` (shipped v0.2.0)

```sql
CREATE TABLE tags (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL UNIQUE,
    color TEXT,
    pinned INTEGER NOT NULL DEFAULT 0,
    sort_order INTEGER NOT NULL DEFAULT 0,
    created_at TEXT NOT NULL
);

CREATE TABLE session_tags (
    session_id TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
    tag_id INTEGER NOT NULL REFERENCES tags(id) ON DELETE CASCADE,
    created_at TEXT NOT NULL,
    PRIMARY KEY (session_id, tag_id)
);
CREATE INDEX idx_session_tags_tag ON session_tags(tag_id);
```

This migration shipped with `projects` + `tag_memories` +

## System prompt assembly

When a session starts or resumes, the server assembles the full
system prompt by concatenating these layers in order:

1. **Base prompt** — short hardcoded string identifying Twrminal.
2. **Tag memories** — content from `tag_memories` for every tag on
   the session, concatenated in **tag sort order** (pinned tags
   first, then ascending `sort_order`, then by id). Tags without a
   matching `tag_memories` row are skipped silently.
3. **Session instructions** — `sessions.session_instructions` if
   set.

Each layer is emitted with a visible header so the assembled
prompt is self-documenting:

```
<!-- layer: base[base] -->
You are Claude running in Twrminal...

<!-- layer: tag_memory[infra] -->
This tag covers home network infrastructure. Relevant directories:
~/infra/, ~/dotfiles/networking/. Prefer nftables over iptables.

<!-- layer: tag_memory[arch-linux] -->
Target platform is Arch Linux with CachyOS kernel. Default to pacman/yay.

<!-- layer: session[session] -->
Focus only on the auth module this session.
```

The assembler runs per turn. v0.2.5 shipped it as a pure function;
v0.2.7 wired `AgentSession` through it.

## Tag memory precedence

Multiple tags on a session may provide conflicting instructions.
Resolution is **document order wins the last write**, where
document order is:

1. Pinned tags before unpinned.
2. Within each group, ascending `tags.sort_order`.
3. Within equal sort orders, ascending tag `id`.

Later layers override earlier ones. **The tag you want to take
precedence should be sorted last.** The user controls this by
reordering tags in the sidebar — no separate priority UI.

**Session instructions always win**, always emitted last.

## Tag defaults (new-session pre-fill)

When a user adds a tag to the new-session form:

- If the tag has `default_working_dir`, the working-dir field
  pre-fills (if it's currently empty, or if only a prior tag's
  default lives there).
- If the tag has `default_model`, the model field pre-fills the
  same way.
- If multiple tags with defaults are selected, the **highest-
  precedence tag wins** — same ordering as tag memories (last in
  sort order, so pinned tags get overridden by later unpinned
  ones if both set a default).
- The user can always manually override; manual edits stick once
  the user has typed into the field.

The pinned tag `twrminal` with `default_working_dir =
~/code/twrminal/` is what replaces "the Twrminal project" —
create the tag once, attach it to new sessions, and the working
dir auto-fills.

## API

### Tags (shipped v0.2.0)

```
GET    /api/tags                             # list, with session counts
POST   /api/tags
GET    /api/tags/{id}
PATCH  /api/tags/{id}
DELETE /api/tags/{id}
```

### Session-tag attachment (shipped v0.2.0)

```
GET    /api/sessions/{id}/tags
POST   /api/sessions/{id}/tags/{tag_id}      # attach (idempotent)
DELETE /api/sessions/{id}/tags/{tag_id}      # detach
```

### Tag memories (shipped v0.2.7)

```
GET    /api/tags/{id}/memory                 # returns content, updated_at (404 if none)
PUT    /api/tags/{id}/memory                 # body: { content }
DELETE /api/tags/{id}/memory
```

### Session system prompt inspect (shipped v0.2.8)

```
GET    /api/sessions/{id}/system_prompt      # { layers: [{ name, kind, content, token_count }], total_tokens }
```

### Session list filtering (shipped v0.2.0 + v0.2.3)

```
GET /api/sessions?tags=1,2&mode=any|all
```

Omitting `tags` returns the full list. `project_id` is **not**
accepted (removed in v0.2.9).

### Session patch (shipped across v0.1.16, v0.2.7)

```
PATCH /api/sessions/{id}    # accepts title, description, max_budget_usd, session_instructions
```

`project_id` is **not** accepted (removed in v0.2.9).

```

```

Click a tag to filter the session list to sessions carrying it.
Any/All toggle controls multi-tag combination. Pinned tags float
to the top.

### New-session form

Gated on at least one attached tag. Shows a tag chip editor
(attach existing / create-and-attach). When a tag with defaults
is attached, working-dir / model fields auto-fill. Submit
disabled until a tag is attached.

### Tag memory editor

Markdown editor with live preview. Reachable from the tag chip
context menu and from the tag's sidebar row. Precedence hint:
*"If multiple tags have conflicting rules, later tags (lower in
the sidebar) override earlier ones."*

### Inspector Context tab (shipped v0.2.8)

Shows the assembled system prompt broken into layers, each with
a color-coded kind badge and approximate token count. Read-only
in v0.2.8; session-instructions inline editor lands in v0.2.12.

### Conversation header badges (shipped v0.2.13)

Memory-stack icon next to model/working_dir, showing count of
active tag memories. Hover lists tag names. Click opens the
Inspector Context tab.

## Agent layer

`agent/prompt.py::assemble_prompt(conn, session_id)` owns
assembly. `AgentSession` calls it before every SDK call. Output
is a single string plus a structured breakdown (layer name, kind,
content, token count) for the Context tab.

No new tools registered. No on-demand loading mechanism. Claude
reads directories and files using the SDK's existing file tools
whenever it needs to, priced and tracked the same as any other
tool call.

## Non-decisions for v0.2.0

- **No projects.** Pinned tags carry the role (see v0.2.9 teardown).
- **No nested tags.** Flat only.
- **No on-demand context injection.** All memories always-inject.
- **No file-backed tag memories.** Content edited in-app, stored in DB.
- **No per-message prompt snapshot storage.** Reconstructable from inputs.
- **No tag memory versioning.** Editing overwrites.
- **No cross-session history rewriting** when tags change. Past turns retain whatever prompt was used at the time.

---

## Gotchas

- **Orphaned tag memories.** Deleting a tag cascades to
  `tag_memories` via FK.
- **Ordering surprise.** Users won't intuit last-tag-wins. The
  one-line hint under the tag memory editor is the mitigation.
- **Prompt assembly is stateless.** Assembled per turn; any tag
  memory edit takes effect on the next prompt without any cache
  invalidation call. Cheap enough to not need caching.
