# Twrminal v0.2.0 — Projects & Tag Memories

Adds two features on top of the v0.1 base: **projects** (optional container for related sessions) and **tag memories** (short markdown instructions injected into the system prompt based on a session's tags).

Design principle: memories are **pointers and rules**, not mirrored document libraries. Claude uses its normal file tools to open directories when it needs to. Keep the system simple.

---

## Summary of changes

- New primitive: **tags** — global, named, with `pinned` + `sort_order`.
  Many-to-many with sessions via `session_tags`. No memory payload yet
  at this layer — that's the next concept. Tags ship first so the
  sidebar panel and session-tagging UI work without any project or
  memory scaffolding.
- New concept: **projects** — optional container giving sessions a
  shared system prompt, working dir, and default model.
- New concept: **tag memories** — per-tag markdown injected into the
  system prompt when that tag is on a session. Always injected when
  present.
- New concept: **layered system prompt** — base + project + tag
  memories + session override, deterministically assembled and
  inspectable.
- Tags remain orthogonal to projects. A session can be in a project
  AND have any tags. Both contribute to the system prompt.
- No breaking changes. Existing sessions get `project_id = NULL`,
  no tags, no session instructions, and behave exactly as before.

---

## Migrations

v0.1 never shipped a tag system. v0.2 introduces tag primitives
first, then adds projects + tag memories on top. Two separate
migrations keep the build order honest and let step 1 land
independently of step 2.

### `0005_tag_primitives.sql`

```sql
-- Tag primitives (new in v0.2)

CREATE TABLE tags (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL UNIQUE,
    color TEXT,
    pinned INTEGER NOT NULL DEFAULT 0,
    sort_order INTEGER NOT NULL DEFAULT 0,
    created_at INTEGER NOT NULL
);

CREATE TABLE session_tags (
    session_id TEXT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
    tag_id INTEGER NOT NULL REFERENCES tags(id) ON DELETE CASCADE,
    created_at INTEGER NOT NULL,
    PRIMARY KEY (session_id, tag_id)
);
CREATE INDEX idx_session_tags_tag ON session_tags(tag_id);
```

### `0006_projects_and_memories.sql`

```sql
-- Projects

CREATE TABLE projects (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL UNIQUE,
    description TEXT,
    system_prompt TEXT,
    working_dir TEXT,
    default_model TEXT,
    pinned INTEGER NOT NULL DEFAULT 0,
    sort_order INTEGER NOT NULL DEFAULT 0,
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL
);

ALTER TABLE sessions ADD COLUMN project_id INTEGER
    REFERENCES projects(id) ON DELETE SET NULL;
CREATE INDEX idx_sessions_project ON sessions(project_id);

-- Tag memories

CREATE TABLE tag_memories (
    tag_id INTEGER PRIMARY KEY REFERENCES tags(id) ON DELETE CASCADE,
    content TEXT NOT NULL,
    updated_at INTEGER NOT NULL
);

-- Per-session system prompt override

ALTER TABLE sessions ADD COLUMN session_instructions TEXT;
```

Both migrations are additive only. No columns dropped, no data
rewritten. Existing v0.1 sessions post-upgrade have `project_id = NULL`
and no tags — behavior unchanged.

## System prompt assembly

When a session starts or resumes, the server assembles the full system prompt by concatenating these layers in order:

1. **Base prompt** — short hardcoded string identifying Twrminal and available tools.
2. **Project prompt** — `projects.system_prompt` if the session has a `project_id`.
3. **Tag memories** — content from `tag_memories` for every tag on the session, concatenated in **tag sort order** (pinned tags first, then by `sort_order`, then by id).
4. **Session instructions** — `sessions.session_instructions` if set.

Each layer is emitted with a visible header so the assembled prompt is self-documenting:

```
<!-- layer: base -->
You are Claude running in Twrminal...

<!-- layer: project[Twrminal] -->
This project is a local Claude Code frontend...

<!-- layer: tag_memory[infra] -->
This tag covers home network infrastructure. Relevant directories:
~/infra/, ~/dotfiles/networking/. Prefer nftables over iptables.

<!-- layer: tag_memory[arch-linux] -->
Target platform is Arch Linux with CachyOS kernel. Default to pacman/yay.

<!-- layer: session_instructions -->
Focus only on the auth module this session.
```

The assembled prompt is cached per session and invalidated when:

- The session's tags change
- The session's project changes
- Any relevant tag memory is updated
- The project's `system_prompt` is updated
- `session_instructions` is edited

## Tag memory precedence

Multiple tags on a session may provide conflicting instructions. Resolution is **document order wins the last write**, where document order is:

1. Pinned tags before unpinned.
2. Within each group, ascending `tags.sort_order`.
3. Within equal sort orders, ascending tag `id`.

Later layers override earlier ones. **The tag you want to take precedence should be sorted last.** The user controls this by reordering tags in the sidebar — no separate priority UI.

**Session instructions always win**, always emitted last.

## Projects — scoping rules

A session gets `project_id` at creation time if a project is currently selected in the UI. It can be changed via `PATCH /api/sessions/{id}` or cleared to `NULL`.

**When a project is selected in the sidebar:**

- The session list filters to that project's sessions.
- The tag panel shows only tags that appear on sessions in that project, with recalculated counts.
- "New session" creates a session pre-assigned to the project, with the project's working dir and default model applied.
- The "All" and "Untagged" views are project-scoped.

**When no project is selected:**

- The full session list is shown.
- All tags are shown with global counts.
- New sessions have `project_id = NULL`.

**Unscoping:** a "Clear project" button returns to the unscoped view. No session data changes.

Directory pointers (e.g., "look in `~/infra/`") live in the project's `system_prompt` as plain markdown. No separate project-files table — Claude opens directories with its normal tools when it needs to.

## API additions

### Projects

```
GET    /api/projects                         # list, with session counts
POST   /api/projects
GET    /api/projects/{id}                    # full object
PATCH  /api/projects/{id}
DELETE /api/projects/{id}                    # sessions become project_id = NULL
```

### Tag memories

```
GET    /api/tags/{id}/memory                 # returns content, updated_at (404 if none)
PUT    /api/tags/{id}/memory                 # body: { content }
DELETE /api/tags/{id}/memory
```

### Session system prompt

```
GET    /api/sessions/{id}/system_prompt      # { layers: [{ name, content, token_count }], total_tokens }
PATCH  /api/sessions/{id}                    # now also accepts session_instructions, project_id
```

### Session list filtering

Existing endpoint extended:

```
GET /api/sessions?project_id=...&tags=1,2&mode=any|all&archived=0
```

`project_id=none` filters to sessions with no project. Omitting the param returns all.

## Agent layer changes

New module `agent/prompt.py` owns assembly. `AgentSession` calls `assemble_prompt(session_id)` before every SDK call. Output is a single string plus a structured breakdown (layer name, content, token count) for the Context tab.

No new tools registered. No on-demand loading mechanism. Claude reads directories and files using the SDK's existing file tools whenever it needs to, priced and tracked the same as any other tool call.

## Non-decisions for v0.2.0

- **No nested projects.** Flat only.
- **No project-files table.** Pointers to directories live in the project's `system_prompt` as plain markdown.
- **No on-demand context injection.** All memories are always-inject. If bloat ever becomes real, revisit in a later version.
- **No file-backed tag memories.** Content is edited in-app and stored in the DB. Source of truth is the DB.
- **No per-message prompt snapshot storage.** The assembled prompt for a given turn isn't persisted; it's reconstructable from inputs.
- **No tag memory versioning.** Editing overwrites.
- **No project-level tags.** Tags remain global.
- **No cross-project history rewriting.** Moving a session changes `project_id`; past turns retain whatever prompt was used at the time.

## Gotchas

- **Orphaned tag memories.** Deleting a tag cascades to `tag_memories` via the FK. Verify cascade fires — add a test.
- **Project deletion.** Sessions become `project_id = NULL`, not deleted. Warn the user but preserve data.
- **Ordering surprise.** Users won't intuit last-tag-wins. The one-line hint under the tag memory editor is the mitigation.
- **Cache invalidation.** The prompt cache is the most likely source of subtle bugs. Every mutation endpoint for projects, tags, session tags, tag memories, and session instructions must call the invalidator. A single `invalidate_session_prompt(session_id)` helper with clear call sites is easier to audit than ad-hoc cache busting.
